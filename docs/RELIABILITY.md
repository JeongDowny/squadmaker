# RELIABILITY.md

## Purpose
이 문서는 입력 부족, 추천 불가, 저장 실패 등 서비스의 실패 상황에서 일관된 동작과 사용자 안내 기준을 정의한다.

이 프로젝트는 복잡한 AI 추론보다 규칙 기반 계산에 가깝지만, 입력 상태와 조합 문제로 인해 충분히 실패 상황이 발생할 수 있다.

## Reliability goals
서비스는 아래를 목표로 한다.

1. 실패하더라도 이유를 설명한다.
2. 가능한 경우 완전 실패보다 부분 결과를 우선 제공한다.
3. 저장 실패 시 데이터 손실을 최소화한다.
4. 수정 후 재계산이 신뢰 가능해야 한다.

## Failure categories

### 1. Input validation failure
입력이 부족하거나 잘못된 경우

예:
- 경기 제목 없음
- 날짜 없음
- 참가 선수 수 부족
- 포지션 미지정 선수 존재
- 쿼터 수 없음
- 쿼터별 포메이션 미선택

대응 원칙:
- 가능한 한 입력 단계에서 즉시 막는다.
- 어느 항목이 문제인지 명확히 표시한다.
- 일반 오류 메시지보다 필드 단위 메시지를 우선한다.

### 2. Lineup generation not possible
현재 입력으로 포메이션 배치가 어려운 경우

예:
- 특정 쿼터 포메이션을 채울 수 있는 선수 구성이 부족함
- 포지션 편중이 너무 심함
- 참가 선수 수 자체가 충분하지 않음

대응 원칙:
- “배치 불가”만 보여주지 않는다.
- 부족한 이유를 구체적으로 설명한다.
- 가능한 대안을 함께 제시한다.

예시 메시지 방향:
- 현재 선수 입력으로 이 포메이션 배치가 어렵습니다.
- 수비 포지션 자원이 부족합니다.
- 다른 포메이션을 선택하거나 참가 선수를 조정해 주세요.

### 3. Partial quality result
완전히 실패는 아니지만 추천 품질이 낮은 경우

예:
- 완전 비적합 포지션이 많이 포함됨
- 쿼터 평균 점수 차이가 큼
- 특정 선수의 출전이 과도하게 몰림

대응 원칙:
- 결과를 아예 숨기지 않는다.
- 결과는 보여주되 품질 이슈를 경고한다.
- 사용자가 수정할 수 있도록 관련 정보를 노출한다.

### 4. Recalculation inconsistency
수동 수정 후 계산 결과가 기대와 다르게 보이는 경우

예:
- 교체 후 출전 쿼터 수가 즉시 안 바뀜
- 평균 점수 반영이 늦음
- 화면과 내부 상태가 어긋남

대응 원칙:
- 계산은 단일 소스 기준으로 수행한다.
- UI 표시용 값과 저장용 값의 계산 기준이 같아야 한다.
- 재계산은 사용자가 명확히 인지할 수 있게 반영한다.

### 5. Save failure
저장 요청이 실패한 경우

예:
- 네트워크 오류
- 인증 만료
- 저장 제한 초과
- 서버 오류

대응 원칙:
- 저장 실패를 침묵 처리하지 않는다.
- 현재 화면 상태는 가능한 한 보존한다.
- 브라우저 임시 저장으로 복구 가능성을 높인다.

사용자 메시지 예시:
- 저장 실패, 브라우저에 임시 저장됨.

## Logged-out reliability
비로그인 사용자는 localStorage에 경기 3개까지 저장 가능하다.

원칙:
- 덮어쓰기 전에 사용자에게 상태를 알린다.
- localStorage 저장 실패 시 가능한 이유를 알려준다.
- 브라우저 데이터가 영구 보관되지 않음을 명시한다.

## Logged-in reliability
로그인 사용자는 경기 3개까지 저장 가능하다.

원칙:
- 저장 가능 개수 초과 시 명확한 안내를 제공한다.
- 기존 경기 수정과 새 경기 저장을 구분한다.
- 인증 만료 시 로그인 재유도 메시지를 보여준다.

## Recommendation fallback policy
추천 계산이 완전하지 않더라도 가능한 범위의 결과를 먼저 제공한다.

우선순위:
1. 완전한 추천 결과
2. 경고를 포함한 부분 결과
3. 배치 불가 이유와 수정 가이드

원칙:
- 아무 것도 안 보여주는 상태를 최소화한다.
- 시스템이 무엇을 알고 있고 무엇을 못 하는지 구분해 알려준다.

## Save recovery policy
저장 실패 시 아래를 우선한다.

1. 현재 편집 상태를 잃지 않는다.
2. 브라우저 임시 저장을 시도한다.
3. 사용자에게 복구 가능 여부를 알린다.

## UI messaging principles
오류 메시지는 아래 원칙을 따른다.

- 짧고 명확하게 쓴다.
- 문제 원인을 가능한 구체적으로 설명한다.
- 사용자가 다음에 무엇을 해야 하는지 알려준다.
- 내부 기술 용어를 그대로 노출하지 않는다.

좋은 예:
- 현재 선수 입력으로 포메이션 배치 불가
- 수비 포지션 선수가 부족합니다
- 다른 포메이션을 선택하거나 참가 선수를 조정해 주세요

나쁜 예:
- calculation failed
- invalid state
- unexpected error

## Consistency rules
- 화면의 선수별 출전 쿼터 수와 저장 값은 같은 기준으로 계산한다.
- 현재 쿼터 평균 점수와 요약 화면 평균 점수는 같은 규칙으로 계산한다.
- 추천 직후 상태와 저장 직전 상태를 혼동하지 않도록 구분 표시를 둔다.

## Observability guidance
중요 이벤트는 개발 및 디버깅 관점에서 추적 가능해야 한다.

예:
- 추천 계산 시작 / 완료
- 추천 불가 이유
- 수동 교체 발생
- 재계산 완료
- 저장 시도 / 성공 / 실패
- localStorage fallback 수행

주의:
- 사용자에게 보이는 메시지와 내부 로깅 메시지는 분리한다.
- 민감하거나 불필요한 데이터는 로그에 남기지 않는다.

## Non-goals
이 문서는 아래를 다루지 않는다.
- 고가용성 인프라 설계
- 실시간 멀티유저 동기화
- 복잡한 분산 장애 대응
