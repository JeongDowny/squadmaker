# ARCHITECTURE.md

## System overview
이 서비스는 축구 경기 운영을 위한 웹 애플리케이션이다.

핵심 구조는 아래와 같다.

- 데이터 저장은 Supabase PostgreSQL을 사용한다.
- ORM은 Prisma를 사용한다.
- 인증은 Google 로그인만 지원한다.
- 추천 계산은 브라우저에서 수행한다.
- 비로그인 사용자는 localStorage 기반으로 임시 저장한다.

즉, 서버는 데이터 저장과 인증을 담당하고, 선발 추천 계산은 클라이언트에서 처리한다.

## Core domain model
이 프로젝트는 팀 중심보다 경기 중심 구조를 사용한다.

핵심 개념은 아래와 같다.

### User
- 로그인 사용자
- 경기 저장 가능
- 저장 가능한 경기 수는 최대 3개

### Player Pool
- 사용자가 보유한 전체 등록 선수 풀
- 경기마다 이 선수들 중 일부를 참가 선수로 선택한다

### Match
- 경기 단위 엔티티
- 제목과 날짜를 가진다
- 쿼터 수를 가진다
- 참가 선수 목록을 가진다
- 최종 수정본 배치를 가진다

### Match Participant
- 특정 경기에 참가하는 선수
- 선수 풀에서 선택된다
- 실제 추천 계산의 대상이 된다

### Quarter
- 경기 내 시간 블록
- 사용자가 직접 개수를 지정한다
- 각 쿼터는 별도 포메이션을 가진다

### Quarter Formation
- 각 쿼터에 선택된 포메이션
- 예: 4-3-3, 4-2-3-1

### Lineup Recommendation
- 특정 경기, 특정 쿼터 집합에 대한 추천 결과
- 현재 v1은 출전 쿼터 수 공평성을 우선하는 단일 추천안이다
- 평균 점수 균형과 포지션 적합성 중심 다중 추천안은 후속 확장 범위다
- 최종 저장 대상은 아니며 계산 결과로 취급한다

### Final Saved Lineup
- 사용자가 추천 결과를 수정한 뒤 저장한 최종본
- 로그인 사용자는 DB 저장
- 비로그인 사용자는 localStorage 저장

## Data flow
전체 흐름은 아래와 같다.

1. 사용자가 선수 풀을 등록한다.
2. 사용자가 경기 제목, 날짜, 쿼터 수를 입력한다.
3. 사용자가 경기 참가 선수를 선택한다.
4. 사용자가 각 쿼터 포메이션을 선택한다.
5. 브라우저에서 추천 계산을 수행한다.
6. 단일 출전 균등 추천안을 생성한다.
7. 사용자가 추천 결과를 보고 쿼터별 포메이션을 다시 조정할 수 있다.
8. 수정 결과에 따라 선수별 출전 쿼터 수를 재계산한다.
9. 최종 수정본을 저장한다.
10. 필요 시 현재 쿼터 또는 전체 쿼터 필드 배치를 PNG로 내보낸다.

## Engineering harness architecture
MVP 초기 단계에서는 실제 기능 구현보다 먼저 하네스 구조를 둔다.

구성은 아래 두 층으로 나눈다.

### Repository-level harness
- 하네스 시나리오와 검증 명령을 단일 데이터 문서로 관리한다.
- 문서 경로, 시나리오 id, 지원 포메이션, 검증 명령 연결 상태를 스크립트로 확인한다.
- 목적은 문서 기준 제품 규칙이 구현 도중 흔들리지 않게 하는 것이다.

### Application-level harness
- `/harness` 화면에서 같은 시나리오를 사람이 읽을 수 있는 UI로 노출한다.
- 각 시나리오는 입력 예시, 기대 동작, 수동 체크리스트, linked docs를 포함한다.
- 목적은 실제 기능이 완성되기 전에도 화면 단위 검증 기준을 고정하는 것이다.

## Why calculation runs in the browser
추천 계산은 시간 복잡도가 비교적 낮고, 사용자가 즉시 결과를 확인해야 한다.

브라우저 계산을 선택한 이유:
- 반응 속도가 빠르다.
- 수정 후 즉시 재계산이 가능하다.
- 서버 부하를 줄일 수 있다.
- 추천 결과가 저장보다 UI 상호작용에 더 가깝다.

## Persistence strategy
### Logged-in users
- Supabase PostgreSQL에 저장
- 경기 최대 3개 저장 가능
- 저장 대상은 최종 수정본 1개

### Logged-out users
- localStorage 사용
- 경기 3개까지 저장 가능
- 로그인하지 않으면 장기 보관을 보장하지 않는다

## Authentication
- Google 로그인만 지원
- 비로그인 상태에서도 사용 가능
- 로그인 시 저장 기능이 활성화된다

## Formation policy
MVP에서는 템플릿 기반 포메이션만 허용한다.

기본 지원 포메이션:
- 4-4-2
- 4-3-3
- 4-2-3-1
- 3-5-2
- 3-4-3
- 4-1-4-1
- 4-5-1
- 4-3-1-2
- 4-1-2-1-2
- 3-4-1-2

각 쿼터는 서로 다른 포메이션을 가질 수 있다.

## Recommendation scoring architecture
현재 v1 추천 계산은 출전 쿼터 수 공평성을 최우선으로 하는 결정적 greedy 방식이다.

현재 v1 처리 순서:
- 선수별 출전 쿼터 수가 적은 선수를 우선한다.
- 같은 출전 수라면 포지션 적합도가 높은 선수를 우선한다.
- 같은 조건이면 입력 순서를 유지해 결과를 안정적으로 만든다.
- GK가 비-GK 슬롯에 먼저 소모되지 않도록 후보를 제한한다.

포지션 적합성은 현재 v1에서 동률 해소와 슬롯 배치 품질을 위해 다음 매핑을 사용한다.
- 주포지션: 1.0
- 부포지션: 0.7
- 인접 계열 포지션: 0.4
- 완전 비적합 포지션: 0.1

후속 다중 추천에서는 평균 점수 균형과 주포지션 일치율을 별도 관점으로 다시 확장할 수 있다.

## Storage shape
DB에 영속 저장되는 핵심 결과는 아래다.

- 경기 제목
- 경기 날짜
- 참가 선수 목록
- 쿼터 수
- 쿼터별 포메이션
- 최종 수정본 선발 배치
- 선수별 출전 쿼터 수 요약

## Reliability considerations
- 포메이션 배치 불가 시 이유를 설명해야 한다.
- 저장 실패 시 브라우저 임시 저장을 사용한다.
- 입력이 부족하면 가능한 범위와 불가능한 이유를 함께 보여준다.
