## PR-001 landing preview refinement and quarter switching mock

### 문제
- 랜딩에 문제 설명, core value, what coaches need fast 같은 설명 섹션이 많아 서비스 예시 화면에 시선이 덜 모였다.
- 예시 전술 보드는 정적인 단일 쿼터 화면이라 1Q~4Q에 따라 포메이션과 선발 배치가 어떻게 달라지는지 보여주지 못했다.
- 출전 요약도 일부 선수만 보여줘 전체 선수의 출전 횟수와 출전 쿼터를 한 번에 확인하기 어려웠다.
- `How It Works` 문구도 서비스 랜딩 톤보다 문서 설명에 가까운 표현이 섞여 있었다.

### 판단 근거
- `new-user-onboarding.md` 기준으로 예시 팀 체험은 설명보다 체험 중심이어야 한다.
- `FRONTEND.md`, `DESIGN.md` 기준으로 결과 화면의 핵심은 쿼터 탭, 필드 배치, 출전 쿼터 수, 평균 점수 순서로 읽혀야 한다.
- 사용자 요청에 따라 이번 수정은 서비스 예시 섹션 강화와 랜딩 카피 톤 조정에 집중하고, 추천 엔진이나 저장 기능은 건드리지 않았다.

### 실행 내용
- `docs/exec-plans/active/service-intro-landing.md`를 갱신해 서비스 예시 중심 정리, 쿼터 탭 전환, 전체 출전 요약 요구를 반영했다.
- `components/landing/service-landing-page.tsx`를 수정해 문제 설명, core value, what coaches need fast 섹션을 제거하고 예시 중심 레이아웃으로 재구성했다.
- 서비스 이름 표기를 `SqaudMaker`로 반영했다.
- `How It Works`와 랜딩 전반의 카피를 문서 설명 톤이 아니라 사용자 안내 톤으로 다시 작성했다.
- `components/landing/mock-lineup-preview.tsx`를 클라이언트 컴포넌트로 바꾸고 1Q~4Q 버튼 전환에 따라 포메이션, 배치, 요약 수치가 바뀌게 했다.
- 전술 보드에서 공격수가 위쪽, 골키퍼가 아래쪽에 배치되도록 방향을 수정했다.
- quarter summary에 전체 선수 15명의 출전 쿼터 수와 출전 쿼터를 모두 표시하고, `GK 고정`, `출전 부족` 같은 특별 태그만 남겼다.

- CLI 환경이라 실제 브라우저 스크린샷 캡처는 수행하지 못했다.

## PR-002 harness screenshot review rule update

### 문제
- UI 변경 작업에서 스크린샷 검증이 필요하다는 규칙은 있었지만, 무엇을 캡처해야 하는지와 무엇을 기준으로 리뷰해야 하는지가 문서상 충분히 구체적이지 않았다.
- 하네스 프로그래밍에 스크린샷 생성, `DESIGN.md` 비교 검토, 정보 위계/CTA/전술 보드 가독성 점검, 수정 반복 기준을 일관되게 적용할 운영 규칙이 필요했다.

### 판단 근거
- `AGENTS.md`는 실제 작업 workflow와 definition of done의 source of truth라 여기서 가장 먼저 규칙을 강화해야 했다.
- 현재 active exec-plan과 MVP 검증 전략도 같은 기준을 따라야 작업마다 스크린샷 검증 방식이 흔들리지 않는다.
- 이번 변경은 문서 정책 강화 작업이므로 코드나 UI는 건드리지 않았다.

### 실행 내용
- `AGENTS.md`에 UI 변경 시 데스크톱/모바일 스크린샷 생성, `docs/DESIGN.md` 비교 검토, 정보 위계/CTA 가시성/전술 보드 가독성 확인, 최소 1회 이상 수정 반복 규칙을 추가했다.
- `docs/product-specs/active/mvp-build.md`의 validation strategy를 같은 기준으로 갱신했다.
- `docs/exec-plans/active/harness-foundation.md`와 `docs/exec-plans/active/service-intro-landing.md`의 screenshot checklist를 새 운영 규칙에 맞게 갱신했다.

### 검증 방법
- 변경된 문서 diff를 직접 확인했다.
- `AGENTS.md`, active exec-plan, MVP validation strategy가 같은 방향으로 정렬됐는지 검토했다.
- 문서 변경만 수행했으므로 별도 lint/typecheck/build는 실행하지 않았다.

### 실행 전 결과
- 스크린샷 검증 규칙이 “확인” 수준으로만 적혀 있어 데스크톱/모바일 캡처 수량, 비교 기준, 수정 반복 기준이 모호했다.

### 실행 후 결과
- UI 변경 하네스 작업 시 필요한 스크린샷 생성과 리뷰 기준이 문서에 더 구체적으로 명시됐다.
- 전역 규칙, MVP 검증 전략, active exec-plan의 스크린샷 체크리스트가 같은 방향으로 정렬됐다.

### 회고 / 남은 이슈
- 현재 규칙은 스크린샷 생성과 수동 리뷰 기준까지는 명확해졌지만, 실제 캡처 도구나 저장 경로 규칙은 아직 정하지 않았다.
- 이후 실제 브라우저 자동화 도입 여부에 따라 스크린샷 생성 방식을 더 구체화할 수 있다.

## PR-004 harness workflow scaffolding on top of current app structure

### 문제
- 추천 구조처럼 실행과 검증을 반복할 수 있는 하네스 운영 계층이 필요했지만, 현재 저장소는 `app/`, `components/`, `lib/` 중심 구조라 바로 그 루프를 걸 수 있는 별도 시나리오/픽스처/리포트 계층이 부족했다.
- 사용자 요청은 전체 `src/` 마이그레이션이 아니라 현 구조를 유지한 채 하네스 프로그래밍에 필요한 뼈대를 붙이는 것이었다.
- 초기 하네스 스크립트를 `tsx` 기반으로 연결했지만, 현재 실행 환경에서는 IPC pipe 권한 문제로 명령이 실패해 실제 반복 루프를 돌릴 수 없었다.

### 판단 근거
- 기존 Next.js 구조를 건드리지 않고 `harness/`, `tests/`, `.codex/`, `scripts/`를 추가하는 쪽이 작업 리스크가 낮고 도입 속도가 빠르다.
- 하네스 프로그래밍의 핵심은 구조 모양보다 실행 가능성이라, 제약 환경에서도 바로 도는 `node` 기반 명령으로 고정하는 쪽이 맞다.
- `AGENTS.md`와 기존 `data/harness/scenarios.json` 흐름을 유지하면서 새 시나리오 JSON과 실행 리포트 계층을 병행하면 현재 하네스와 새 운영 루프를 연결할 수 있다.

### 실행 내용
- `.codex/config.toml`, `.codex/agents/*`, `.codex/skills/*`를 추가해 planner / implementer / reviewer 역할과 UI review, regression, release check 가이드를 스캐폴딩했다.
- `harness/scenarios/*`, `harness/fixtures/*`, `harness/evals/*`, `harness/reports/*`를 추가해 실행-검증 시나리오, 예시 데이터, 체크리스트, 최신/히스토리 리포트 구조를 만들었다.
- `tests/unit`, `tests/integration`, `tests/e2e`, `tests/visual` 디렉토리를 추가해 테스트 확장 위치를 고정했다.
- `scripts/validate.mjs`, `seed-harness-data.mjs`, `capture-screenshots.mjs`, `summarize-report.mjs`를 추가하고 package scripts를 모두 `node` 기준으로 다시 연결했다.
- `harness/evals/validate-scenarios.mjs`를 추가해 새 시나리오 JSON의 필수 필드와 linked doc / exec-plan / fixture 경로를 검증하게 했다.
- `scripts/validate.mjs`에서 기존 `data/harness` 검증과 새 하네스 스캐폴드 검증을 함께 수행하도록 묶었다.
- `README.md`, `docs/index.md`, `docs/exec-plans/active/harness-workflow-scaffolding.md`를 갱신해 새 구조와 명령, 현재 도입 범위를 문서화했다.
- 불필요해진 `tsx` 의존성을 제거했다.

### 검증 방법
- `npm run harness:repo`
- `npm run harness:scenarios`
- `npm run harness:seed`
- `npm run harness:capture`
- `npm run harness:report`
- `npm run test`
- `npm run lint`
- `npm run typecheck`
- `npm run build`

### 실행 전 결과
- 현재 구조를 유지한 채 반복 가능한 하네스 자산 위치가 정리돼 있지 않았다.
- `.codex/agents`, `.codex/skills`, `harness/`, `tests/` 스캐폴딩이 없어 역할 분리와 시나리오 기반 검증 루프를 일관되게 잡기 어려웠다.
- `tsx` 기반 하네스 스크립트는 이 환경에서 실행되지 않아 명령이 실제 루프로 동작하지 못했다.

### 실행 후 결과
- 기존 앱 구조를 유지한 채 `.codex/`, `harness/`, `tests/`, `scripts/` 기반 하네스 운영 계층이 추가됐다.
- `npm run harness:repo`, `harness:scenarios`, `harness:seed`, `harness:capture`, `harness:report`, `test`, `lint`, `typecheck`, `build`가 모두 통과했다.
- `harness/reports/latest/`에 seed summary와 screenshot capture plan이 생성돼 실행 결과를 바로 확인할 수 있게 됐다.
- 저장소 수준 검증에서 기존 `data/harness`와 새 하네스 스캐폴드 29개 경로를 함께 확인하도록 묶였다.

### 회고 / 남은 이슈

## PR-005 src migration, harness source unification, and Playwright test adoption

### 문제
- 앱 코드는 루트 `app/`, `components/`, `lib/`에 있었고, 하네스 메타데이터는 `data/harness/scenarios.json`와 `harness/scenarios/*`로 나뉘어 있어 source of truth가 이중화돼 있었다.
- `test`는 메타데이터 검증만 수행하고 실제 브라우저에서 랜딩과 `/harness` 화면을 검증하지 못했다.
- UI 변경 시 스크린샷 검증 규칙은 있었지만, 실제 저장소 루프 안에서 Playwright 캡처가 연결돼 있지 않았다.

### 판단 근거
- 저장소 규모가 아직 작아서 지금 `src/` 구조로 정리하면 feature 단위 확장이 쉬워지고, 루트 `harness/`는 실행 자산 전용 계층으로 더 명확해진다.
- `/harness` 화면도 루트 하네스 문서를 직접 읽어야 repository/application harness가 같은 데이터를 본다고 말할 수 있다.
- 하네스 프로그래밍의 핵심은 문서, 시나리오, 실제 화면 검증이 같은 루프에 있어야 한다는 점이므로, `test`에 Playwright를 포함시키는 것이 맞다.

### 실행 내용
- 앱 코드를 `src/app`, `src/features/landing`, `src/features/harness`로 이동하고 import 경로와 TypeScript alias를 `src` 기준으로 정리했다.
- `data/harness/scenarios.json`를 `harness/config/repository-harness.json`로 이전하고, `/harness` 화면이 이 루트 manifest를 직접 읽도록 `src/features/harness/server/harness-document.ts`를 작성했다.
- `package.json`에 `start:playwright`, `test:playwright`, `test:e2e`, `test:visual`을 추가하고 `test`가 하네스 validator 뒤에 Playwright를 실행하도록 바꿨다.
- `playwright.config.ts`를 추가해 built app을 `next start`로 띄우고, 데스크톱/모바일 Chromium 프로젝트를 실행하도록 설정했다.
- `tests/e2e/landing.spec.ts`, `tests/e2e/harness-dashboard.spec.ts`, `tests/visual/landing-screenshots.spec.ts`를 추가해 랜딩과 `/harness` 라우트를 브라우저에서 검증하고 데스크톱/모바일 랜딩 스크린샷을 생성하게 했다.
- `.github/workflows/harness-ci.yml`를 갱신해 Playwright 브라우저 설치와 `npm run test` 실행을 포함했다.
- `README.md`, `docs/index.md`, `harness/README.md`, `docs/exec-plans/tech-debt-tracker.md`를 갱신해 `src` 구조, 루트 하네스 manifest, Playwright 흐름을 문서화했다.
- `.gitignore`에 Playwright 런타임 산출물과 최신 스크린샷 산출물을 추가해 저장소 노이즈를 줄였다.

### 검증 방법
- `npm run harness:repo`
- `npm run harness:seed`
- `npm run lint`
- `npm run typecheck`
- `npm run test`
- `npm run build`
- `npm run harness:report`

### 실행 전 결과
- 하네스 앱 화면과 저장소 validator가 서로 다른 파일 집합을 기준으로 움직였다.
- `test`가 실제 브라우저 검증을 포함하지 않아 랜딩과 `/harness`의 렌더링/가시성 문제를 자동으로 잡지 못했다.
- UI 스크린샷 규칙이 문서에만 있고, 저장소 명령으로 바로 실행되지는 않았다.

### 실행 후 결과
- 앱 코드는 `src/` 아래 feature 단위 구조로 정리됐고, 루트 `harness/config/repository-harness.json`가 repository/application harness의 공용 source of truth가 됐다.
- `npm run test`에서 하네스 validator와 Playwright 브라우저 검증이 함께 실행된다.

- `npm run harness:report`에서 최신 스크린샷 파일까지 함께 요약된다.
- lint, typecheck, build, full test가 모두 통과했다.

### 회고 / 남은 이슈
- 현재 Playwright visual test는 스크린샷 생성까지는 자동화했지만, baseline diff나 디자인 규칙 자동 점수화까지는 하지 않는다.
- 추천 엔진, 저장, 내보내기 기능이 아직 없어서 하네스 시나리오 중 setup / recommendation / persistence는 문서 계약 수준에 머물러 있다.
- 문서와 진행 상황을 기준으로 다음 내용이 더 필요하다.
  - recommendation용 golden fixture 세트
  - 저장/내보내기 실패 분기용 persistence fixture와 Playwright 시나리오
  - screenshot baseline 관리 규칙 또는 `DESIGN.md` 체크리스트를 구조화한 visual review rubric
