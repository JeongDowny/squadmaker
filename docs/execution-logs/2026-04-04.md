## PR-001 execution log and prompt log scaffolding

### 문제
- 현재 저장소는 `docs/execution-logs/`만 운영하고 있어 무엇을 구현했는지는 남지만, 어떤 사용자 지시와 작업용 재정리 프롬프트에서 출발했는지는 추적하기 어려웠다.
- 하네스 프로그래밍 관점에서는 결과 로그뿐 아니라 입력 로그도 함께 남겨야 다음 작업자가 같은 맥락을 복원할 수 있다.

### 판단 근거
- `AGENTS.md` 기준으로 비사소한 변경이므로 관련 문서 확인, exec-plan, 검증 전략, execution log 기록이 필요했다.
- `docs/ARCHITECTURE.md`와 하네스 문서 기준으로 저장소 운영 규칙은 문서와 실행 명령이 함께 연결돼야 한다.
- prompt log는 execution log를 대체하는 것이 아니라 입력을 보강하는 별도 계층으로 두는 쪽이 기존 운영 방식과 충돌이 적었다.

### 실행 내용
- `docs/exec-plans/active/prompt-log-scaffolding.md`를 추가해 목표, 범위, 명령 설계, 검증 전략을 문서화했다.
- `docs/prompt-logs/README.md`와 `docs/prompt-logs/2026-04-04.md`를 추가해 prompt log 디렉터리, 템플릿, 오늘자 실제 기록을 만들었다.
- `scripts/scaffold-prompt-log.mjs`와 `package.json`의 `harness:prompt-log` 명령을 추가해 날짜 파일과 PR 섹션 템플릿을 자동 생성할 수 있게 했다.
- `AGENTS.md`, `README.md`, `docs/index.md`, `harness/README.md`, `harness/config/repository-harness.json`, `.codex/config.toml`, `.codex/skills/release-check/SKILL.md`를 갱신해 prompt log 정책과 연결 경로를 반영했다.
- `scripts/validate.mjs`에 새 문서/스크립트 경로를 추가해 하네스 스캐폴드 검증에 prompt log 자산을 포함시켰다.

### 검증 방법
- `npm run harness:repo`
- `npm run harness:scenarios`
- `npm run lint`
- `npm run typecheck`
- `npm run harness:prompt-log -- --help`

### 실행 전 결과
- 비사소한 변경의 입력 프롬프트를 남기는 전용 문서 구조와 명령이 없었다.
- execution log와 연결되는 prompt 기록 규칙이 `AGENTS.md`와 README 계층에 없었다.

### 실행 후 결과
- `docs/prompt-logs/` 구조와 템플릿이 추가돼 사용자 원문 프롬프트와 작업용 정리 프롬프트를 날짜 / PR 단위로 기록할 수 있게 됐다.
- `npm run harness:prompt-log` 명령으로 새 prompt log 섹션을 빠르게 스캐폴딩할 수 있게 됐다.
- execution log와 prompt log를 서로 연결하는 운영 규칙이 문서와 하네스 구조에 반영됐다.

### 회고 / 남은 이슈
- 현재 prompt log는 자동 수집이 아니라 명시적 기록 방식이라 실제 프롬프트 텍스트는 사용자가 붙이거나 스크립트 인자로 넘겨야 한다.
- 민감 정보 마스킹은 아직 운영 규칙 수준이며 자동화돼 있지 않다.
- 이후 필요하면 release check나 CI에서 prompt log 누락을 더 강하게 확인할 수 있다.

## PR-002 match editor first screen and auto-create flow

### 문제
- 랜딩의 `시작하기` CTA가 실제 서비스 화면으로 이어지지 않아, 사용자가 첫 편집 화면을 경험하지 못했다.
- 현재 앱에는 `/matches/:matchId` 구조와 경기 자동 생성 흐름이 없어 경기 중심 모델을 체감하기 어려웠다.
- 추천 구현 전 단계에서도 필드 우선 레이아웃, 선수 카드 입력 구조, 다음 단계 recommendation CTA가 먼저 정리돼 있어야 했다.

### 판단 근거
- `AGENTS.md`, `FRONTEND.md`, `DESIGN.md` 기준으로 첫 실제 화면은 긴 폼보다 필드 배치와 선수 관리가 먼저 읽혀야 했다.
- `ARCHITECTURE.md`와 `DOMAIN.md` 기준으로 이 서비스는 경기 중심 모델을 유지해야 하고, 추천 계산 입력이 될 경기 제목/날짜/참가 선수 구조를 먼저 가져야 했다.
- `new-user-onboarding.md` 기준으로 랜딩의 대표 진입점 중 `내 경기 만들기`는 실제 앱 화면으로 짧게 이어져야 했다.
- 추천 계산, 저장, 로그인, DB 연동은 이번 범위가 아니므로 mock CTA와 상태 구조까지만 구현하는 쪽이 맞았다.

### 실행 내용
- `docs/exec-plans/active/match-editor-first-screen.md`를 추가해 범위, UI 방향, 검증 전략, 스크린샷 체크리스트를 고정했다.
- `src/app/matches/new/page.tsx`를 추가해 랜딩 CTA가 `/matches/new`를 거쳐 자동 생성된 `/matches/:matchId`로 redirect되게 했다.
- `src/features/matches/lib/match-editor.ts`에 오늘 날짜 기준 기본 경기 제목/날짜, match id 생성, 포지션 정렬, quarter formation 템플릿, 초기 mock player pool 구조를 추가했다.
- `src/app/matches/[matchId]/page.tsx`와 `src/features/matches/components/match-editor-screen.tsx`를 추가해 server page + client editor 구조로 경기 편집 화면을 구현했다.
- 경기 화면에서 경기 제목/날짜 수정, 필드 우선 전술 보드, 1Q~4Q 포메이션 탭, 오른쪽 선수 풀, 인라인 선수 카드 추가, 주포지션/부포지션/실력 점수 입력, recommendation placeholder 패널을 구현했다.
- `src/features/landing/components/service-landing-page.tsx`의 hero / 하단 CTA를 실제 `시작하기` 링크로 연결했다.
- `tests/e2e/match-editor.spec.ts`와 `tests/visual/match-editor-screenshots.spec.ts`를 추가하고, 기존 landing spec을 새 CTA 흐름에 맞게 갱신했다.
- `scripts/validate.mjs`와 `harness/config/repository-harness.json`를 갱신해 새 route, component, test, active exec-plan, setup target 경로를 하네스 검증에 반영했다.
- `react-hook-form` 의존성을 추가해 문서 기준의 폼 구조를 실제 구현에 맞췄다.

### 검증 방법
- `npm run lint`
- `npm run typecheck`
- `npm run harness:repo`
- `npm run harness:scenarios`
- `npm run test:e2e`
- `npm run test:visual`
- `npm run harness:report`
- 수동 확인 기준으로는 `시작하기 -> /matches/:matchId 진입`, 경기 제목/날짜 기본값, 제목/날짜 수정, 선수 추가, 선수 카드 정렬, recommendation placeholder 노출 여부를 Playwright 시나리오에 반영했다.

### 실행 전 결과
- 랜딩 CTA는 mock 상태였고 실제 경기 편집 화면으로 이어지지 않았다.
- `/matches/:matchId` 라우트와 경기 초안 생성 흐름이 없었다.
- 필드 우선 레이아웃, 선수 카드 반복 입력, 포지션 정렬 구조가 아직 코드에 없었다.

### 실행 후 결과
- 랜딩의 `시작하기`로 자동 생성된 `/matches/:matchId` 화면에 진입할 수 있게 됐다.
- 경기 제목은 오늘 날짜 기준 `YYYY-MM-DD 경기`, 경기 날짜는 오늘 날짜로 기본 설정된다.
- 경기 제목/날짜 수정, 선수 카드 인라인 추가, 주포지션/부포지션/실력 점수 입력, 포지션 우선순위 정렬이 동작한다.
- 필드가 먼저 보이고 선수 풀이 옆에 보이는 데스크톱 레이아웃과 모바일 세로 흐름이 모두 Playwright e2e/visual 검증을 통과했다.
- `harness/reports/latest/screenshots/landing-desktop.png`, `landing-mobile.png`, `match-editor-desktop.png`, `match-editor-mobile.png`가 생성됐다.
- 스크린샷 생성 결과와 구현 구조 기준으로 정보 위계는 필드 -> 선수 풀 -> recommendation placeholder 순으로 유지됐고, 랜딩 `시작하기` CTA 가시성과 전술 보드 슬롯 가독성에도 문제를 발견하지 못했다.

### 회고 / 남은 이슈
- 현재 필드 보드는 recommendation 전 단계의 빈 슬롯 준비 화면이며, 아직 선수를 슬롯에 실제 배치하지는 않는다.
- 선수 카드에는 삭제나 참가/후보 분기 같은 편집 기능을 넣지 않았고, 다음 단계 recommendation 연결에 필요한 최소 입력 구조만 남겼다.
- `match-setup-validation` 하네스 시나리오가 기대하는 필드 단위 오류 메시지와 추천 진입 차단 규칙은 아직 구현되지 않았으므로 이후 setup hardening에서 이어서 보강해야 한다.

## PR-003 match editor density and service-like layout refinement

### 문제
- 현재 경기 배치 준비 화면은 정보량은 충분했지만 상단 정보 바와 설명 문구의 비중이 커서 실제 서비스 편집 화면보다 mock 시안처럼 보였다.
- 필드와 우측 선수 패널의 비율이 다소 느슨하고, 우측 영역도 관리 패널보다 입력 카드 나열 인상이 강했다.
- 추천 placeholder가 별도 카드로 크게 노출돼 필드 중심 정보 위계를 조금 흐리고 있었다.

### 판단 근거
- `AGENTS.md`, `FRONTEND.md`, `DESIGN.md` 기준으로 이 화면은 설명보다 필드 배치와 선수 관리가 먼저 읽혀야 한다.
- active exec-plan의 refinement note도 상단 경기 정보 압축, 필드/우측 패널 비율 조정, 카드 밀도 개선을 다음 단계 우선 과제로 명시하고 있었다.
- 사용자가 제공한 STARTING 11 레퍼런스는 브랜드 복제가 아니라 필드 우선 구조, 우측 관리 패널 밀도, 압축된 편집 화면 톤만 참고 대상으로 삼는 것이 맞았다.
- 사용자가 지정한 `docs/exec-plans/active/mvp-build.md`는 저장소에 없어서, 실제 존재하는 관련 active plan인 `docs/exec-plans/active/match-editor-first-screen.md`를 기준 exec-plan으로 사용했다.

### 실행 내용
- `src/features/matches/components/match-editor-screen.tsx`를 리팩터링해 상단을 압축형 편집 바로 줄이고, 활성 쿼터/입력 완료 상태를 작은 메트릭으로 재배치했다.
- 메인 레이아웃을 `필드 중심 캔버스 + 우측 관리 패널` 비율로 다시 잡고, 추천 placeholder를 필드 하단의 보조 스트립으로 내려 존재감을 낮췄다.
- 필드 패널은 전술 보드 헤더, 쿼터 칩, 보드 상단 상태 pill을 추가해 실제 운영 중인 편집 화면처럼 빠르게 읽히도록 조정했다.
- 우측 선수 영역은 카드 패딩과 라벨 밀도를 줄이고, 상태 chip과 summary metric을 넣어 입력 카드 나열보다 현재 경기 선수 관리 패널에 가깝게 정리했다.
- `tests/e2e/match-editor.spec.ts`, `tests/visual/match-editor-screenshots.spec.ts`를 새 헤딩 텍스트에 맞게 갱신했다.
- `docs/exec-plans/active/match-editor-first-screen.md`에 이번 밀도 리팩토링 방향과 수동 체크 항목을 반영했다.
- 기존 스크린샷을 `harness/reports/latest/screenshots/match-editor-desktop-before-refactor.png`, `match-editor-mobile-before-refactor.png`로 보존한 뒤 새 visual 캡처를 생성했다.

### 검증 방법
- `npm run lint`
- `npm run typecheck`
- `npm run test:e2e`
- `npm run test:visual`
- 스크린샷 비교:
  - before: `harness/reports/latest/screenshots/match-editor-desktop-before-refactor.png`

### 실행 전 결과
- 상단 경기 정보 영역이 크고 설명성 문구가 많아 서비스 화면보다 시안처럼 읽혔다.
- 필드와 우측 패널의 중심축이 지금보다 약했고, 추천 placeholder가 비교적 크게 보였다.
- 우측 선수 리스트는 실제 운영 패널보다 입력 카드 묶음 인상이 강했다.

### 실행 후 결과
- 상단 영역이 더 작고 압축적인 편집 바로 정리돼 경기 이름/날짜/활성 쿼터를 빠르게 수정하는 구조가 됐다.
- 데스크톱 기준 필드 보드와 우측 선수 패널이 화면 중심으로 더 선명하게 보이고, 필드 하단 placeholder는 보조 톤으로 낮아졌다.
- 우측 패널은 summary metric, 상태 chip, 더 촘촘한 입력 행으로 구성돼 “현재 경기 선수 관리” 패널 인상이 강화됐다.

- `npm run lint`, `npm run typecheck`, `npm run test:e2e`, `npm run test:visual`이 모두 통과했다.

### 회고 / 남은 이슈
- 현재는 시각 구조 리팩토링 단계라 선수 삭제, 참가/후보 상태 전환, 추천 진입 전 validation 메시지 같은 관리 기능은 아직 최소 수준이다.
- screenshot review는 생성 결과 기준으로 정보 위계, CTA 가시성, 전술 보드 가독성, 모바일 세로 흐름을 점검했고, 이번 턴에서는 추가 수정 반복이 필요할 만큼의 문제는 발견하지 못했다.

## PR-004 landing page service-like SaaS refinement

### 문제
- 현재 랜딩페이지는 전술 보드 예시 자체는 괜찮았지만, hero에서 대상 사용자와 핵심 가치가 충분히 빠르게 읽히지 않아 mock 소개 페이지처럼 느껴졌다.
- CTA가 주로 하단에 몰려 있고, preview도 결과 화면 showcase라기보다 상대적으로 긴 앱 dump처럼 읽히는 구간이 있었다.
- 설명 섹션 구조는 유지할 수 있었지만, 여백과 카드 밀도, typography hierarchy가 실제 서비스 랜딩 톤으로 더 다듬어질 필요가 있었다.

### 판단 근거
- `new-user-onboarding.md` 기준으로 첫 방문 사용자는 복잡한 설명보다 서비스 목적과 첫 행동을 빠르게 이해해야 한다.
- `DESIGN.md`와 `FRONTEND.md` 기준으로 SquadMaker는 깔끔한 SaaS 인상과 결과 화면 중심 흐름을 동시에 유지해야 한다.
- 사용자가 제공한 Starting11 레퍼런스는 브랜드 복제가 아니라 hero 구조, CTA 배치, 제품 미리보기 방식, 전체 섹션 리듬만 참고 대상으로 삼는 것이 맞았다.
- 사용자가 지정한 `docs/exec-plans/active/mvp-build.md`는 저장소에 없어서, 실제 관련 active plan인 `docs/exec-plans/active/service-intro-landing.md`를 실행 기준으로 사용했다.

### 실행 내용
- `src/features/landing/components/service-landing-page.tsx`를 리팩토링해 hero를 대상 사용자, 핵심 가치, 진입 CTA 중심의 압축된 SaaS 구조로 다시 짰다.
- hero 안에 quick-start 카드와 핵심 가치 bullet을 넣어 첫 화면에서 “무엇을 위한 서비스인지”와 “어디로 들어가야 하는지”가 더 빨리 읽히게 만들었다.
- 문제 정의, preview, 흐름, CTA 섹션의 카드 밀도와 여백, 제목 계층을 조정해 mock 소개 페이지보다 실제 서비스 랜딩에 가까운 리듬으로 재배치했다.
- preview 중간에 `/matches/new`로 이어지는 `내 경기 만들기` CTA를 추가하고, 하단 CTA도 같은 진입 흐름에 맞춰 유지했다.
- `src/features/landing/components/mock-lineup-preview.tsx`를 리팩토링해 긴 설명을 줄이고 필드, 추천안, 출전 요약이 함께 보이는 핵심 장면 중심 showcase로 압축했다.
- `tests/e2e/landing.spec.ts`를 새 hero/CTA 문구에 맞게 갱신했고, `tests/visual/landing-screenshots.spec.ts`는 새 레이아웃 캡처 기준으로 유지했다.
- `docs/exec-plans/active/service-intro-landing.md`에 hero/CTA refinement 방향과 추가 수동 체크 항목을 반영했다.
- 기존 랜딩 스크린샷을 `harness/reports/latest/screenshots/landing-desktop-before-refactor.png`, `landing-mobile-before-refactor.png`로 보존한 뒤 새 visual 캡처를 생성했다.

### 실행 전 결과
- 첫 화면에서 서비스 대상, 핵심 가치, 시작 CTA가 지금보다 느리게 읽혔다.
- CTA 위계가 하단 중심이라 preview 흐름 중간에서 실제 행동으로 전환되는 느낌이 약했다.
- preview와 설명 섹션이 조금 더 mock showcase처럼 보였고, 전체 랜딩 리듬도 지금보다 덜 정돈돼 있었다.

### 실행 후 결과
- hero에서 대상 사용자, 문제, 핵심 가치, `/matches/new` 진입 CTA가 첫 화면 안에서 더 빠르게 읽히는 구조가 됐다.
- preview는 긴 화면 dump보다 핵심 장면 중심으로 압축됐고, 중간 CTA와 하단 CTA가 모두 자연스럽게 `/matches/new`로 이어지게 정리됐다.
- 문제 정의, 설명, 흐름, CTA 섹션의 카드 밀도와 간격이 정돈돼 전체 랜딩이 실제 서비스 소개 페이지에 더 가까운 분위기로 바뀌었다.
- before/after 산출물의 파일 크기와 해시가 모두 달라졌고, 데스크톱 스크린샷은 `1238017 -> 1152455` bytes, 모바일 스크린샷은 `3596965 -> 3800423` bytes로 갱신돼 시각 변화가 명확히 반영됐다.
- `npm run lint`, `npm run typecheck`, `npm run test:e2e`, `npm run test:visual`이 모두 통과했다.
