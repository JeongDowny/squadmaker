# 2026-04-01

## PR-001 repository/application harness foundation

### 문제
- 제품 문서는 정리되어 있지만 저장소와 앱 안에 이를 반복 검증할 하네스 구조가 없었다.
- 기본 `create-next-app` 초기 화면만 있어 제품 규칙과 구현 기준이 코드에 드러나지 않았다.

### 판단 근거
- `AGENTS.md` 기준으로 비사소한 변경이므로 계획, 검증 전략, execution log가 필요했다.
- `docs/PRODUCT_SENSE.md`, `docs/ARCHITECTURE.md`, `docs/FRONTEND.md` 기준으로 제품 규칙과 모바일 사용성 기준을 먼저 구조화해야 했다.
- `docs/product-specs/active/mvp-build.md`에 이미 "문서 기반 하네스 구조 정착"이 MVP 범위로 명시되어 있었다.

### 실행 내용
- `docs/exec-plans/active/harness-foundation.md`를 추가해 목표, 범위, 테스트 전략, 스크린샷 체크리스트를 문서화했다.
- `data/harness/scenarios.json`에 repository/application harness 공용 시나리오 source of truth를 추가했다.
- `scripts/validate-harness.mjs`와 `package.json` 스크립트를 추가해 저장소 레벨 검증 명령을 만들었다.
- `/` 랜딩 페이지를 제품/하네스 진입점으로 교체하고 `/harness` 대시보드를 추가했다.
- `README.md`, `docs/ARCHITECTURE.md`, `docs/FRONTEND.md`, `docs/product-specs/active/mvp-build.md`를 현재 하네스 구조에 맞게 갱신했다.

### 검증 방법
- `npm run harness:repo`
- `npm run test`
- `npm run typecheck`
- `npm run lint`
- `npm run build`
- 기본 Turbopack build는 샌드박스 포트 바인딩 오류가 있어 `build` 스크립트를 webpack 경로로 고정했다.
- UI 스크린샷 검증 항목은 exec-plan에 포함했지만, 이번 CLI 환경에서는 실제 스크린샷 캡처를 수행하지 못했다.

### 실행 전 결과
- 저장소에 하네스 시나리오 데이터와 검증 스크립트가 없었다.
- 랜딩 페이지는 Next.js 기본 템플릿 상태였다.
- `/harness` 경로와 하네스 관련 문서 연결 구조가 없었다.

### 실행 후 결과
- 저장소에서 제품 시나리오와 linked docs를 JSON 기반으로 검증할 수 있게 됐다.
- 랜딩 페이지에서 제품 규칙과 하네스 진입점을 바로 확인할 수 있게 됐다.
- `/harness` 화면에서 repository-level/application-level harness 정보를 카드 기반으로 읽을 수 있게 됐다.
- 기본 `npm run build`도 webpack 경로에서 성공하도록 정리했다.

### 회고 / 남은 이슈
- 현재 application harness는 시나리오 가이드 대시보드 수준이며, 실제 기능 상태를 실시간 반영하지는 않는다.
- 브라우저 기반 스크린샷 검증과 상호작용 검증은 이후 실제 UI 구현 단계에서 보강해야 한다.
- 추천 엔진, 편집기, 저장 기능이 생기면 같은 하네스 데이터를 테스트 fixture와 demo seed로 확장하는 것이 다음 단계다.

## PR-002 harness rule hardening and handoff contract

### 문제
- 하네스 규칙은 있었지만 문서 충돌, failure path 누락, handoff 조건 부족 때문에 운영 규칙으로 쓰기엔 약했다.
- `test`가 실제 기능 테스트처럼 읽히지만 현재는 하네스 메타데이터 검증뿐이라 의미가 모호했다.

### 판단 근거
- `AGENTS.md` 기준으로 비사소한 변경은 더 강한 계획/검증/기록 구조가 필요했다.
- `docs/RELIABILITY.md`, `docs/product-specs/team-save-and-export.md`, `docs/product-specs/quarter-lineup-editor.md`에 이미 failure path와 edge case 규칙이 있었지만 하네스에 반영되지 않았다.
- `docs/ARCHITECTURE.md`와 `docs/product-specs/formation-recommendation.md` 사이 추천 가중치 표현이 구현자에게 혼선을 줄 수 있었다.

### 실행 내용
- 하네스 exec-plan에 handoff 계약, failure path, 테스트 의미 구분 규칙을 추가했다.
- `data/harness/scenarios.json`에 `scenarioType`, `linkedExecPlan`, `inputs.notes`, `handoff` 필드를 추가하고 setup/recommendation/reliability failure 시나리오를 확장했다.
- `scripts/validate-harness.mjs`를 강화해 stage, scenarioType, exec-plan 경로, handoff 필수 필드, 명령 category, 날짜 형식 등을 검증하게 했다.
- `/harness` 카드 UI에 scenario type, input notes, handoff 섹션을 노출했다.
- `docs/ARCHITECTURE.md`, `docs/FRONTEND.md`, `docs/product-specs/active/mvp-build.md`를 강화된 하네스 규칙에 맞게 갱신했다.
- `package.json`에 `test:harness`를 추가해 현재 `test`의 의미를 더 명확히 했다.

### 검증 방법
- `npm run harness:repo`
- `npm run test:harness`
- `npm run test`
- `npm run typecheck`
- `npm run lint`
- `npm run build`

### 실행 전 결과
- 하네스 시나리오에 handoff 정보가 없었다.
- validator는 파일 존재와 중복 id 정도만 검증했다.
- failure path와 edge case가 하네스에서 거의 다뤄지지 않았다.

### 실행 후 결과
- 하네스가 happy path, warning path, failure path를 모두 다루게 됐다.
- 각 시나리오에 다음 작업자가 바로 이어받을 handoff 계약이 포함됐다.
- repository validator가 더 강한 게이트 역할을 하도록 보강됐다.
- 추천 가중치의 source of truth가 architecture 문서에서도 명확해졌다.

### 회고 / 남은 이슈
- 현재 `test:harness`는 여전히 메타데이터/구조 검증 중심이며, 실제 기능 테스트 도입은 다음 단계다.
- 하네스 시나리오와 실제 추천/저장 구현이 생기면 fixture 또는 demo data로 연결하는 작업이 필요하다.
- 스크린샷 자동화와 상호작용 테스트는 아직 수동 검증 중심이다.

## PR-003 harness prioritization, ownership, status, and CI gate

### 문제
- 하네스 시나리오가 있어도 무엇부터 구현할지, 누가 맡는지, 현재 상태가 어떤지 바로 읽히지 않았다.
- expected result와 verification target이 구조화되어 있지 않아 실제 구현/검증 연결이 약했다.
- CI gate가 없어 하네스 명령 집합이 저장소 운영 규칙으로 강제되지 않았다.

### 판단 근거
- 하네스를 효과적으로 쓰려면 시나리오가 단순 요구사항이 아니라 작업 우선순위와 완료 기준까지 담아야 했다.
- `/harness` 화면이 운영 보드 역할을 하려면 status/owner/priority를 직접 보여줘야 했다.
- 저장소 레벨에서 실행하는 명령이 CI에서도 동일하게 강제되어야 handoff 품질이 유지된다.

### 실행 내용
- `data/harness/scenarios.json`에 `priority`, `owner`, `status`, `expectedResults`, `verification`, `ciGate`를 추가했다.
- `lib/harness.ts`와 `/harness` 대시보드 UI를 업데이트해 우선순위 순 정렬, status 표시, expected result, verification target을 노출했다.
- `scripts/validate-harness.mjs`를 확장해 새 메타데이터와 CI workflow 존재 여부를 검증하게 했다.
- `.github/workflows/harness-ci.yml`을 추가해 `harness:repo`, `test:harness`, `typecheck`, `lint`, `build`를 실행하도록 했다.
- `README.md`, `docs/FRONTEND.md`, `docs/exec-plans/active/harness-foundation.md`를 새 운영 방식에 맞게 갱신했다.

### 검증 방법
- `npm run harness:repo`
- `npm run test:harness`
- `npm run test`
- `npm run typecheck`
- `npm run lint`
- `npm run build`

### 실행 전 결과
- 시나리오에 우선순위, owner, status, expected result 구조가 없었다.
- `/harness`에서 구현 상태를 운영 보드처럼 읽기 어려웠다.
- CI workflow가 없어 하네스 명령 세트가 로컬 규칙에만 머물렀다.

### 실행 후 결과
- 시나리오별 구현 우선순위와 소유권, 현재 상태를 하네스 데이터와 UI에서 직접 확인할 수 있게 됐다.
- expected result와 verification target이 구조화되어 구현/검증 연결점이 분명해졌다.
- CI gate가 하네스 명령 세트를 자동 실행하도록 추가됐다.

### 회고 / 남은 이슈
- verification target은 현재 existing/planned 수준의 연결이며, 실제 테스트 파일과 runtime assertion 연결은 다음 단계다.
- scenario status는 아직 대부분 `contract-ready`라 실제 기능 구현이 시작되면 지속적으로 갱신하는 운영 습관이 필요하다.
- 스크린샷 자동 비교는 아직 CI에 포함되지 않았다.

## PR-004 root landing messaging alignment

### 문제
- 루트 페이지가 제품 소개보다 하네스 소개를 더 강하게 드러내고 있어 첫 방문자가 서비스 목적을 빠르게 이해하기 어려웠다.
- 홈 화면 하단에 하네스 대시보드가 직접 노출돼 랜딩의 초점이 제품 가치보다 구현 보조 구조로 기울어 있었다.

### 판단 근거
- `docs/FRONTEND.md`는 `/`를 랜딩과 시작 CTA 역할로 정의하고 있고, `/harness`는 별도 검증 대시보드로 분리돼야 한다.
- `docs/product-specs/new-user-onboarding.md` 기준으로 첫 화면은 서비스 목적, 첫 결과 흐름, 예시 체험 또는 시작 동선을 빠르게 이해시키는 데 집중해야 한다.
- 이번 수정은 추천 로직이나 저장 정책이 아니라 홈 화면 정보 구조와 메시지 정렬 작업이므로 기존 제품 문서와의 정합성이 핵심이었다.

### 실행 내용
- `app/page.tsx`에서 하네스 중심 문구를 서비스 소개 문구로 교체했다.
- 히어로 영역을 제품 가치와 첫 결과 3단계 흐름 중심으로 재구성했다.
- 서비스 핵심 가치와 제품 우선순위를 별도 섹션으로 분리했다.
- 홈 화면에서 `HarnessDashboard` 직접 노출을 제거하고 `/harness` 링크만 보조 안내로 남겼다.

### 검증 방법
- `npm run lint`
- `npm run typecheck`
- `npm run test`
- `npm run build`
- 스크린샷 검증은 CLI 환경 한계로 실제 캡처를 수행하지 못했고, 모바일 대응은 기존 반응형 레이아웃 패턴 유지로 점검했다.

### 실행 전 결과
- 홈 화면 상단 문구가 "MVP Foundation", "repository/application harness" 중심으로 읽혔다.
- 하네스 대시보드가 루트 페이지 본문에 직접 렌더링돼 제품 랜딩과 개발용 대시보드의 경계가 약했다.

### 실행 후 결과
- 루트 페이지가 감독과 운영자를 위한 서비스 소개, 핵심 가치, 제품 규칙 설명에 집중하도록 정리됐다.
- 첫 방문자는 홈에서 서비스 흐름을 이해하고, 구현용 하네스는 필요할 때만 `/harness`로 진입하게 됐다.
- 정적 검증과 빌드가 모두 통과해 메시지 재구성이 기존 앱 구조를 깨지 않음을 확인했다.

### 회고 / 남은 이슈
- 현재 홈 화면에는 실제 "예시 팀 체험"이나 "내 경기 만들기" 경로가 아직 없어, CTA는 설명형 앵커와 하네스 보조 링크 수준에 머물러 있다.
- 이후 실제 온보딩 경로가 생기면 루트 CTA를 문서상의 대표 진입점에 맞춰 다시 연결해야 한다.

## PR-005 service intro landing page

### 문제
- 루트 페이지에 서비스 소개는 있었지만, 첫 방문자가 문제 정의, 핵심 가치, 사용 흐름, 결과 화면 예시를 한 번에 이해하기에는 정보 구조가 부족했다.
- `new-user-onboarding.md`에서 요구하는 예시 팀 체험 성격의 랜딩 경험과 전술 보드형 결과 미리보기가 아직 없었다.

### 판단 근거
- `AGENTS.md` 기준으로 비사소한 UI 변경이므로 active exec-plan, 검증 계획, execution log가 필요했다.
- `FRONTEND.md`는 `/`를 랜딩 / 예시 팀 체험 / 시작 CTA 역할로 정의하고 있고, `DESIGN.md`는 결과 예시 화면을 잔디 느낌의 전술 보드로 보이게 하라고 규정한다.
- `PRODUCT_SENSE.md`, `ARCHITECTURE.md`, `DOMAIN.md` 기준으로 랜딩은 공평성, 쿼터 평균 점수 균형, 포지션 적합성, 쿼터별 포메이션 우선 흐름을 일관되게 드러내야 했다.

### 실행 내용
- `docs/exec-plans/active/service-intro-landing.md`를 추가해 목표, 범위, UI 방향, 테스트 전략, 스크린샷 체크리스트를 문서화했다.
- `components/landing/service-landing-page.tsx`를 추가해 문제 정의, 핵심 가치, 사용 흐름, CTA 섹션을 랜딩 전용 컴포넌트로 분리했다.
- `components/landing/mock-lineup-preview.tsx`를 추가해 쿼터 탭, 추천안 A/B/C, 전술 보드형 필드, 선수 배치, 출전 쿼터 수 요약을 담은 static mock 예시 화면을 구현했다.
- `app/page.tsx`를 얇은 진입점으로 바꾸고 랜딩 전용 컴포넌트를 연결했다.

### 검증 방법
- `npm run lint`
- `npm run typecheck`
- `npm run test`
- `npm run build`
- CLI 환경에서는 실제 브라우저 스크린샷 캡처를 수행하지 못해 시각 검증은 코드 구조와 빌드 결과 중심으로 확인했다.

### 실행 전 결과
- 랜딩에 결과 예시 화면이 없었고, 문제/가치/흐름 설명이 제품 온보딩용 구조로 충분히 정리돼 있지 않았다.

- 루트 페이지에 문제 정의, 핵심 가치, 사용 흐름, 예시 결과 화면, CTA가 모두 포함된 서비스 소개 랜딩이 추가됐다.

- lint, typecheck, harness test, build가 모두 통과했다.

### 회고 / 남은 이슈
- 실제 브라우저에서 모바일/데스크톱 스크린샷 검증은 아직 수행하지 못했다.
- CTA는 현재 앵커 기반이며, 이후 예시 팀 체험 라우트와 경기 생성 라우트가 생기면 실제 진입점으로 교체해야 한다.
