## PR-001 matches manual placement editor refinement

### 문제
- `/matches/:matchId` 화면에 랜딩 복귀 버튼, 현재 편집 상태, 활성 쿼터/입력 완료/실력 점수 metric 등 보조 정보가 많아 필드 편집 화면보다 준비용 mock 화면처럼 보였다.
- 선수 패널이 이름/주포지션만 빠르게 보는 구조가 아니라 세부 입력 카드가 기본으로 크게 노출됐다.
- 필드 슬롯에는 `추천 대기` placeholder가 보여 실제 선수를 배치하고 수정하는 결과 편집 화면으로 읽히지 않았다.

### 판단 근거
- `AGENTS.md`, `FRONTEND.md`, `DESIGN.md` 기준으로 matches 화면은 필드 배치가 최우선이고 모바일에서도 같은 핵심 편집이 가능해야 한다.
- `quarter-lineup-editor.md` 기준으로 데스크톱에서는 드래그 앤 드롭을 보조 수단으로 지원할 수 있고, 모바일에서는 클릭/탭 기반 대체 흐름이 필요하다.

- 추천 결과가 나온 뒤 포메이션을 바꿔도 단일 추천 v1 정책이 명확하지 않았다.
- 랜딩에 긴 문제/설명 섹션과 중복 헤더가 있었고, 경기 관리/팀 관리 네비게이션이 없었다.

### 실행 후 결과
- matches 추천 버튼은 `출전 균등 추천` 단일 CTA로 동작한다.
- 실력 점수 입력과 A/B/C 추천안 UI가 제거됐다.
- 선수 목록은 이름, 주포지션, 이 경기 출전 쿼터 수, 참가 여부를 우선 표시한다.
- 참가 체크를 끈 선수는 추천 필드에 배치되지 않는다.
- 추천 후 쿼터 포메이션을 바꾸면 같은 참가 명단 기준으로 필드 슬롯과 선수 배치가 다시 계산된다.
- 공통 네비게이션에서 `경기 관리`, `팀 관리`, `새 경기`로 이동할 수 있다.
- `/matches`와 `/players`에서 로컬 상태 기반 생성/수정/삭제 UI가 동작한다.
- 랜딩은 짧은 서비스 설명, 스크린샷형 흐름 목업, 마지막 경기 관리 CTA 중심으로 읽힌다.
- 스크린샷 기준 정보 위계는 랜딩에서 설명 -> 목업 -> CTA, matches에서 쿼터 설정 -> 필드 -> 선수 목록 순서로 유지됐다.

### 회고 / 남은 이슈

- 현재 경기/팀 관리 화면은 요청 범위에 맞춰 로컬 UI 상태만 사용하며, 실제 저장/로그인/DB/localStorage 영속 저장은 후속 범위다.
- 평균 점수 균형과 A/B/C 다중 추천을 다시 도입하려면 실력 점수 입력, 저장 shape, 추천 카드 UI, 문서를 함께 재설계해야 한다.

## PR-004 persistent workspace and Supabase bridge

### 문제
- 경기/팀 관리가 local state만 사용해 새로고침 후 데이터가 사라졌다.
- 경기 목록에서 별도 편집 버튼을 눌러야 했고, 목록 카드 자체가 편집 진입점으로 동작하지 않았다.
- 팀 관리 선수와 경기 내 선수 관리가 분리돼 있어 팀 선수 풀을 경기 참가자로 재사용할 수 없었다.
- 추천 결과가 계산값으로만 존재해 사용자가 추천 이후 필드 배치를 직접 수정하고 저장하기 어려웠다.
- Supabase/Prisma 의존성, schema, auth route, 환경변수 구조가 아직 없었다.

### 판단 근거
- 사용자 선택 기준에 따라 localStorage 경기는 3개까지 저장하도록 정책을 변경했다.
- 팀 선수 정보는 항상 source of truth로 공유하고, 경기별로는 참가 여부와 `lineups`만 보존하는 방식이 현재 UI 요구에 가장 단순하다.
- Supabase 실제 연동은 Supabase Auth Google과 Prisma DB write를 결합하는 방식으로 결정했다.
- Supabase 환경변수가 없는 로컬/CI에서도 localStorage 기능과 테스트가 깨지지 않아야 한다.

### 실행 내용
- `src/features/persistence/lib/local-store.ts`를 추가해 선수 풀과 경기 목록 localStorage read/write, 경기 생성, 팀 선수 병합, lineups 정규화를 구현했다.
- `MatchEditorDraft`에 `lineups`를 추가하고, 추천 결과를 `lineups`로 저장하도록 바꿨다.
- 경기 편집 화면에서 localStorage 저장 match와 팀 선수 풀을 읽어 초기화하고, 변경사항을 localStorage에 자동 저장하게 했다.
- 선수 목록과 필드 슬롯에 HTML5 drag-and-drop을 추가해 추천 후 슬롯 배치를 바꿀 수 있게 했다.
- 클릭 fallback으로 슬롯 선택 후 선수 선택 배치를 지원했다.
- 참가 제외 시 해당 선수를 모든 lineups에서 제거하게 했다.
- 경기 관리 화면을 localStorage 기반으로 바꾸고, 카드 클릭으로 `/matches/:matchId`에 진입하게 했다.
- 팀 관리 화면을 localStorage 기반으로 바꾸고, 새 경기에서 팀 선수가 선수 관리에 표시되게 했다.
- `@supabase/ssr`, `@supabase/supabase-js`, `prisma`, `@prisma/client`를 추가했다.
- `prisma/schema.prisma`, Supabase server/proxy/auth callback, Google 로그인/로그아웃 서버 액션, Prisma snapshot 저장/불러오기 서버 액션을 추가했다.
- `.env.example`에 Supabase/Prisma 환경변수 이름을 추가했다.
- localStorage 3경기 정책에 맞게 `AGENTS.md`, `ARCHITECTURE.md`, `FRONTEND.md`, `SECURITY.md`, `RELIABILITY.md`, `team-save-and-export.md`, `mvp-build.md`, `PLANS.md`를 갱신했다.
- E2E 테스트에 localStorage 복원, 경기 카드 클릭 진입, 팀 관리와 새 경기 연동, 필드 드래그 배치 수정을 추가했다.

### 검증 방법
- `npx prisma generate`
- `npm run lint`
- `npm run typecheck`
- `npm run test:e2e`

### 실행 전 결과
- `/matches`와 `/players` 데이터는 새로고침 시 사라졌다.
- `/matches`에서 리스트 카드 클릭으로 편집 화면에 들어갈 수 없었다.
- `/players`에서 만든 선수가 `/matches/new`의 경기 내 선수 관리에 연결되지 않았다.
- 추천 결과 필드 배치는 사용자가 직접 저장 가능한 상태로 수정할 수 없었다.
- Supabase/Prisma 관련 파일과 의존성이 없었다.

### 실행 후 결과
- 팀 선수와 경기 목록이 localStorage에 저장되고 새로고침 후 복원된다.
- localStorage 경기는 최대 3개까지 만들 수 있고, 초과 시 안내한다.
- 경기 카드의 `카드 눌러 수정` 영역을 누르면 해당 경기 편집 화면으로 이동한다.
- 팀 관리에서 만든 선수는 새 경기 편집 화면의 선수 관리에 표시된다.
- 추천 생성 후 필드 슬롯을 드래그해 선수 배치를 swap할 수 있고, 출전 쿼터 수는 lineups 기준으로 다시 계산된다.
- Supabase 환경변수가 없을 때도 localStorage 플로우는 동작하며, 클라우드 불러오기/저장은 설정 또는 로그인 필요 메시지를 반환한다.
- Prisma schema와 Supabase Auth/서버 액션 골격이 추가돼 환경변수와 Supabase 프로젝트 설정 후 DB 저장 경로를 사용할 수 있다.

### 회고 / 남은 이슈
- 최초 `npx prisma generate`는 Prisma engine cache의 권한 문제로 실패해 승인된 권한으로 재실행했다.
- 실제 Supabase 프로젝트 URL, anon key, DATABASE_URL, Google provider 설정은 이 환경에 없어서 live DB 저장은 실행하지 못했다.
- 현재 클라우드 저장은 수동 `클라우드 저장`/`클라우드 불러오기` 버튼 기반이다. 로그인 후 자동 동기화와 충돌 해결 정책은 후속으로 고도화할 수 있다.
- PNG 내보내기는 여전히 후속 범위다.
