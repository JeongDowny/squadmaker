# SECURITY.md

## Purpose
이 문서는 이 프로젝트의 인증, 저장, 입력 검증, 비밀값 관리와 관련된 보안 원칙을 정의한다.

이 서비스는 금융·의료처럼 고민감 데이터를 다루지는 않지만, 로그인 정보, 저장 데이터, 브라우저 저장 데이터, 사용자 입력값을 안전하게 처리해야 한다.

## Security goals
이 프로젝트의 보안 목표는 아래와 같다.

1. 인증 정보와 서버 비밀값을 안전하게 관리한다.
2. 로그인 사용자 데이터와 비로그인 임시 데이터를 명확히 구분한다.
3. 사용자 입력값을 그대로 신뢰하지 않는다.
4. 불필요한 개인정보 수집을 최소화한다.
5. 저장 및 내보내기 기능이 의도하지 않은 데이터 노출로 이어지지 않게 한다.

## Authentication policy
- 로그인 방식은 구글 로그인만 지원한다.
- 로그인하지 않아도 서비스 핵심 기능은 사용할 수 있다.
- 로그인은 저장 기능과 경기 관리 기능을 위해 사용한다.
- 서버에서 인증된 사용자만 DB 저장 기능을 사용할 수 있다.

## Authorization policy
- 사용자는 자신의 경기 데이터만 저장, 조회, 수정할 수 있어야 한다.
- 다른 사용자의 경기 데이터에 접근하는 기능은 MVP에 포함하지 않는다.
- 공유 링크 기반 공개 접근은 지원하지 않는다.
- 관리자 전용 기능은 MVP 범위에 포함하지 않는다.

## Data classification
이 프로젝트의 데이터는 아래처럼 구분한다.

### 1. Public-like low-sensitivity input
- 경기 제목
- 경기 날짜
- 선수 이름
- 포지션
- 실력 점수
- 포메이션 선택 결과

이 정보는 일반적으로 고도의 민감정보는 아니지만, 사용자 소유 데이터이므로 무단 접근을 막아야 한다.

### 2. Authentication-related data
- 사용자 계정 식별 정보
- 세션 정보
- 인증 토큰 관련 정보

이 정보는 가장 엄격하게 다뤄야 한다.

### 3. Local temporary data
- 비로그인 상태의 경기 1개
- localStorage에 저장되는 입력값 및 최종 결과

이 정보는 브라우저 환경에 저장되므로 영구 보관을 보장하지 않는다.

## Input validation policy
사용자 입력은 항상 검증한다.

검증 대상 예시:
- 경기 제목 길이
- 선수 이름 길이
- 포지션 값이 허용 목록 안에 있는지
- 실력 점수가 1~10 정수인지
- 쿼터 수가 유효한 숫자인지
- 쿼터별 포메이션 값이 허용 목록 안에 있는지

원칙:
- 프론트엔드 검증과 서버 검증을 함께 둔다.
- 클라이언트 검증만으로 보안이 충분하다고 가정하지 않는다.
- 문자열 입력은 항상 길이 제한을 둔다.
- 예상하지 않은 값은 거부한다.

## Sanitization policy
이 서비스는 텍스트 입력을 많이 다루지 않지만, 아래 원칙을 따른다.

- 사용자 입력 문자열은 그대로 HTML로 렌더링하지 않는다.
- 위험한 마크업이나 스크립트 삽입을 허용하지 않는다.
- 저장 전에 기본적인 sanitize와 trim 처리를 적용한다.
- 내보내기용 이미지에도 사용자 입력이 직접 UI를 깨지 않게 길이 제한과 표시 규칙을 적용한다.

## Secret management
- Supabase 관련 비밀값은 서버 전용 환경 변수로 관리한다.
- 클라이언트에 노출 가능한 공개 키와 서버 전용 키를 혼동하지 않는다.
- 서비스 키나 데이터베이스 비밀값을 클라이언트 코드에 두지 않는다.
- `.env` 파일은 커밋하지 않는다.
- 예시 문서에는 실제 비밀값을 적지 않는다.

## Storage security policy

### Logged-out users
- 비로그인 사용자는 localStorage에 경기 1개만 저장할 수 있다.
- localStorage 데이터는 보안 저장소가 아니므로 민감 데이터 저장 용도로 사용하지 않는다.
- localStorage 데이터는 사용자의 브라우저 환경에 종속된다.
- 비로그인 저장은 임시 편의 기능으로 취급한다.

### Logged-in users
- 로그인 사용자 경기 데이터는 DB에 저장한다.
- 로그인 사용자는 최대 경기 3개까지 저장할 수 있다.
- 각 경기 데이터는 소유자 기준으로 접근 제한이 있어야 한다.

## Save policy
- 저장 요청은 인증 상태를 확인한 뒤 수행한다.
- 저장 가능 개수 제한은 서버에서도 검증한다.
- 저장 실패 시 현재 화면 상태를 잃지 않도록 해야 한다.
- 저장 실패 시 브라우저 임시 저장 fallback을 허용한다.

## Export policy
- 이미지 내보내기는 PNG만 지원한다.
- 내보내기 대상은 필드 배치만 포함한다.
- 공유 링크 기능은 지원하지 않는다.
- 외부에 바로 퍼블릭하게 노출되는 저장 기능은 제공하지 않는다.

## Logging policy
로그에는 아래를 남길 수 있다.
- 추천 계산 시작/완료
- 저장 시도/성공/실패
- 인증 여부
- 추천 불가 이유의 코드화된 정보

로그에 남기지 말아야 할 것:
- 비밀 키
- 세션 토큰
- 액세스 토큰
- 민감한 인증 세부값
- 불필요하게 긴 원본 사용자 입력 전체

## Error message policy
사용자에게 보여주는 오류 메시지는 다음 원칙을 따른다.
- 내부 스택 트레이스를 노출하지 않는다.
- 서버 구현 세부사항을 그대로 드러내지 않는다.
- 사용자가 다음에 무엇을 해야 하는지 알려준다.

좋은 예:
- 저장에 실패했습니다. 잠시 후 다시 시도해 주세요.
- 로그인 상태를 확인한 뒤 다시 저장해 주세요.

피해야 할 예:
- database insert failed at relation...
- auth token invalid, internal exception...

## Dependency policy
- 인증, 저장, export 관련 라이브러리는 신뢰 가능한 패키지를 사용한다.
- 장기간 방치된 패키지는 도입 전에 검토한다.
- drag-and-drop, export 라이브러리는 최소 권한 원칙에 맞게 사용한다.

## Non-goals
이 문서는 아래를 다루지 않는다.
- 기업용 고급 접근 제어
- 세분화된 역할 기반 권한 체계
- 공개 협업 링크 모델
